## Phase 2 Bug & Concern Log

### Bugs Found and Fixed

**1. Windows Timer Resolution â€” SQLite Pagination Collapse**
- **File:** `internal/destination/sqlite.go`
- **What happened:** Used `time.Now().UnixNano()` as the fallback row ID when a payload had no `id` field. Windows timer resolution is ~100ns, so a tight loop inserting 10 rows gave many of them identical nanosecond timestamps. `INSERT OR REPLACE` collapsed them to a single row. `TestSQLitePagination` got `1 0 0 0` instead of `3 3 3 1`.
- **Fix:** Replaced with an `atomic.Int64` counter producing `auto-1`, `auto-2`, etc. â€” guaranteed unique regardless of clock resolution.

**2. Module Path Mismatch**
- **What happened:** All files were generated with module path `github.com/shawnsammartano/bubblefish`. Your actual repo is `github.com/shawnsammartano-hub/BubbleFish`. This caused Go to try fetching a nonexistent remote module.
- **Fix:** Global find-and-replace across all `.go` files and `go.mod`. Required two passes â€” once for the wrong username, once for the case mismatch (`bubblefish` vs `BubbleFish`) that Windows's case-insensitive filesystem exposed.

---

### Concerns Worth Watching

**3. WAL Rotation â€” Filename Collision Risk**
- Rotated segments are named `wal-{timestamp}.jsonl` with second precision. If rotation fires twice within the same second (possible under a very low `maxBytes` threshold in tests or a hammered prod system), the second `os.Rename` will silently overwrite the first archived segment, losing data.
- **Recommendation:** Add a monotonic counter suffix: `wal-{timestamp}-{n}.jsonl`.

**4. WAL Rotation Log Output**
- The test output showed the same filename printed 10 times during `TestRotation`. The rotation is working correctly â€” the timestamp-based name just doesn't change within a single test run. Not a data loss risk, but the stderr noise could mask real rotation errors in CI logs.

**5. Worker Retry Backoff Is Wall-Clock Slow**
- `TestWorkerRetryTransient` takes 3 seconds because the real backoff schedule (`1s, 2s, 4s`) runs in tests. No test helper exists to inject a faster schedule.
- **Recommendation:** Make `backoffSchedule` injectable via `WorkerConfig` before Phase 3 so tests don't slow down as the suite grows.

**6. SQLite Single-Writer Constraint**
- `db.SetMaxOpenConns(1)` is correct for SQLite, but it means the destination will serialize all writes. Under high queue throughput this will become the bottleneck before the queue or WAL does.
- **Not a Phase 2 issue**, but worth noting before Phase 3 wires the worker pool to SQLite at scale.

**1. `queue.Enqueue` type mismatch**
The daemon originally defined its own `queueItem` struct and tried to pass it to `queue.Enqueue()`, which only accepts `queue.Item`. I caught and fixed this â€” but if anything got out of sync during the edits, this will be the first compile error you see. **2. Phantom directory created during build**
When I ran `mkdir -p` with brace expansion in the sandbox, it literally created a folder named `{wal,queue,idempotency,destination}` alongside the real directories. That junk folder ended up in the zip/tar archives, which likely contributed to the corruption. Your downloaded files are clean â€” but if you extracted the archives before they were confirmed corrupt, check for a strangely named folder in `internal/` and delete it.

## đźźˇ Functional Concerns

**3. Worker doesn't start until `Run()` is called**
`NewDaemon()` constructs everything but the background worker goroutine only starts inside `Run()`. This means if you call `ServeHTTP` directly in tests without calling `Run()`, payloads will be accepted and WAL-written but never reach SQLite. The acceptance test for "payload appears in SQLite after write" will pass in production but could silently fail in unit tests. A small `time.Sleep` or channel sync in the test would fix it.

**4. `ServeHTTP` rebuilds the router on every call**
`d.ServeHTTP()` calls `d.buildRouter()` which reconstructs the entire chi router and all middleware on every single HTTP request. Fine for tests, real performance problem under load. The router should be built once in `NewDaemon()` or `Run()` and stored on the struct.

**5. Rate limit bucket missing for sources with no `rate_limit` config**
If a source config omits `rate_limit` entirely (both `requests_per_second` and `burst` are zero), no bucket is created and that source gets unlimited throughput. This is the intended behavior per the code comment, but it means a misconfigured source file silently bypasses rate limiting with no warning logged.

---

## đźź  Edge Cases

**6. `MaxBytesReader` error detection is fragile**
The 413 response relies on `io.ReadAll` returning an error when the body exceeds the limit. `MaxBytesReader` does return an error in this case, but the error message check isn't explicit â€” any read error (network drop, client disconnect) will also return 413 instead of a more appropriate 400/500. Low risk for internal use, worth hardening later.

**7. Idempotency key only deduplicates when the header is present**
If no `Idempotency-Key` header is sent, every request is treated as unique regardless of identical body content. The comment in the code acknowledges this. It's by design but means the "duplicate idempotency_key returns 200 without re-writing" acceptance criteria only works if the client sends the header â€” the spec's PowerShell test doesn't send one, so that test won't exercise dedup.

**8. `GET /query/{dest}` uses the source name from auth context to apply response filters**
The filter lookup does `h.d.sources[source]` where `source` comes from the auth context (the API key owner). If you query with an admin token instead of a source key, `source` will be empty, no filter config will be found, and results will be returned unfiltered. Probably fine, but worth knowing.
