# BubbleFish Community — Phased Build Plan

**Version:** 2.0 | **Date:** March 2026 | **Author:** Shawn Sammartano | **CONFIDENTIAL**

## HOW TO USE THIS DOCUMENT

This build plan is designed for AI-assisted development. Each phase is a self-contained unit you can hand to an AI coding tool (Claude Code, Cursor, etc.) as a single prompt. Each phase includes:

- **CONTEXT:** What exists from previous phases (paste this to orient the AI)
- **DELIVERABLES:** Exact files to create or modify
- **ACCEPTANCE CRITERIA:** How to verify the phase is complete
- **DO NOT:** Guardrails to prevent scope creep

**Start from scratch.** Your old Phase 1 code (the Node.js prototype and the early Go skeleton) is obsolete. The v2 architecture is different enough that retrofitting would take longer than rebuilding. Clean repo, clean start.

**One phase per AI session.** Do not try to build multiple phases in one session. Each phase is scoped to fit within a single extended AI conversation with room for debugging.

## PHASE 2: WAL + QUEUE + DESTINATION INTERFACE
**Duration:** 3-4 days | **Depends on:** Phase 1

### CONTEXT
Phases 0-1 complete. You have: working CLI with `init` and `build`, build pipeline that validates TOMLs and writes compiled JSON, all config structs, env var resolution, example templates. No runtime components exist yet.

### DELIVERABLES

**1. internal/wal/wal.go:** Implement WAL interface.
- JSONL format: one JSON-serialized TranslatedPayload per line
- Mutex-protected Append()
- Each entry includes schema_version
- Replay() reads from offset, returns slice of TranslatedPayload
- LastOffset() returns current byte offset

**2. internal/wal/rotation.go:** Size-based auto-rotation.
- Check segment size after each Append()
- If segment exceeds WALMaxSegmentMB from daemon config, rotate
- Rotation: close current file, rename to wal-{timestamp}.jsonl, open new wal-current.jsonl
- Old segments are not deleted (manual cleanup in Community, automated in Enterprise)

**3. internal/wal/replay.go:** Schema-version-aware replay.
- If entry.SchemaVersion == CurrentSchemaVersion: return as-is
- If migration function registered: apply and return
- If no migration: log warning, skip entry, continue

**4. internal/queue/queue.go:** Implement Queue interface.
- Buffered channel (capacity = load_shed_threshold from daemon config)
- Enqueue returns error if channel full (load shedding)
- Stats() returns current counts

**5. internal/queue/worker.go:** Worker goroutine.
- Reads from channel, calls Destination.Write()
- On error: check ClassifiedError.Class
  - TRANSIENT: retry with exponential backoff (1s, 2s, 4s), max 3 attempts
  - PERMANENT: log and drop
  - AUTH: log and drop (should not happen at write time)
- On success: update completed count

**6. internal/queue/circuit.go:** CircuitBreaker.
- Closed → Open after N consecutive TRANSIENT failures (configurable)
- Open → HalfOpen after RecoveryInterval
- HalfOpen → Closed on success, Open on failure
- AllowRequest() checked before each Write()

**7. internal/destination/plugin.go:** Destination interface definition (from Technical Spec section 4.1).

**8. internal/destination/sqlite.go:** SQLiteDestination implementation.
- Uses modernc.org/sqlite (pure Go)
- Init(): create table if not exists (id TEXT PRIMARY KEY, timestamp TEXT, source TEXT, content TEXT, metadata TEXT, vector BLOB, schema_version TEXT, idempotency_key TEXT)
- Write(): INSERT OR REPLACE
- Read(): SELECT with optional filters, limit, offset
- HealthCheck(): SELECT 1
- Close(): close DB connection

**9. internal/idempotency/store.go:** In-memory idempotency store.
- map[string]time.Time guarded by sync.RWMutex
- Exists() checks if key present and not expired
- Store() adds key with current time
- Background goroutine prunes expired keys every 60 seconds
- Default dedup window: 5 minutes (configurable via daemon.toml in future)

### ACCEPTANCE CRITERIA
- [ ] WAL: Append 1000 payloads, kill process, Replay() recovers all 1000
- [ ] WAL: Segment rotates at configured size, new segment starts correctly
- [ ] WAL: Replay skips entries with unknown schema version, logs warning
- [ ] Queue: Enqueue returns error when channel full
- [ ] Queue: Worker retries TRANSIENT errors 3 times with backoff
- [ ] Queue: Worker drops PERMANENT errors after logging
- [ ] Circuit breaker: Opens after 5 consecutive failures, half-opens after 30s
- [ ] SQLite: Write + Read round-trip works
- [ ] SQLite: HealthCheck returns nil when DB is accessible
- [ ] Idempotency: Exists() returns true for recently stored key, false after expiry
- [ ] `go build ./...` still succeeds, `go vet` passes

### DO NOT
- Do not implement OpenBrain, ChromaDB, or Postgres destinations yet
- Do not implement HTTP server
- Do not implement TUI or Web UI
- Do not implement transforms (execution) yet

## PHASE 3: DAEMON + HTTP HANDLERS + TRANSFORMS
**Duration:** 4-5 days | **Depends on:** Phase 2

### CONTEXT
Phases 0-2 complete. You have: build pipeline, WAL with rotation and replay, queue with circuit breaker, SQLite destination, idempotency store. No HTTP server exists yet.

### DELIVERABLES

**1. internal/daemon/daemon.go:** Daemon struct and lifecycle.
```go
type Daemon struct {
    config       *config.CompiledConfig
    wal          wal.WAL
    queues       map[string]queue.Queue        // key = destination name
    destinations map[string]destination.Destination
    idempotency  idempotency.IdempotencyStore
    httpServer   *http.Server
    logger       *observability.Logger
}

func NewDaemon(configDir string) (*Daemon, error)  // Load compiled config, init everything
func (d *Daemon) Run() error                        // Start HTTP, start workers, block
func (d *Daemon) Shutdown(ctx context.Context) error // Graceful: stop HTTP, drain queues, close WAL
```

**2. internal/daemon/handlers.go:** HTTP route handlers.
- POST /inbound/{endpoint}: Full write path as specified in Technical Spec section 6.1
  1. Match endpoint to compiled source
  2. Validate API key (Authorization: Bearer header)
  3. Check payload size against max_payload_bytes
  4. Apply field mapping (routing.ApplyMapping)
  5. Apply transforms (transform.Apply)
  6. Check idempotency (if key present)
  7. Check rate limit
  8. Append to WAL
  9. Check load shedding
  10. Enqueue
  11. Return 200 with payload ID
- GET /query/{dest}: Read path as specified in Technical Spec section 6.2
- GET /api/status: Return JSON status object (Technical Spec section 6.3)
- GET /api/logs: Return recent log entries

**3. internal/daemon/middleware.go:**
- AuthMiddleware: validates API key, maps to source identity
- RateLimitMiddleware: per-source token bucket, returns 429 with Retry-After

**4. internal/routing/mapper.go:**
```go
func ApplyMapping(rawJSON []byte, mappings []config.FieldMapping) (config.TranslatedPayload, error)
```
- Uses util/jsonpath.go to extract fields
- Sets ID (uuid), Timestamp (now), SchemaVersion (current)
- Returns error if required mapping cannot be extracted

**5. internal/transform/engine.go + functions.go + parser.go:**
```go
func Apply(payload config.TranslatedPayload, rules []config.TransformRule) (config.TranslatedPayload, error)
```
- Supported functions: trim(field), concat(field1, field2, ...), coalesce(field1, default), conditional(condition, ifTrue, ifFalse)
- Parser validates expression syntax
- Engine evaluates expressions against payload fields

**6. internal/presenter/filter.go:**
```go
func ApplyFilter(data []map[string]interface{}, filterCfg config.ResponseFilterConfig) []map[string]interface{}
```
- If include_fields set: return ONLY those fields
- If exclude_fields set: return everything EXCEPT those fields
- include_fields takes precedence if both set

**7. internal/security/auth.go:**
- ValidateAPIKey(key string, sources map[string]CompiledSource) (sourceName string, error)
- Maps API key to exactly one source. If key matches no source or multiple sources, reject.

**8. internal/observability/logger.go:** Structured JSON logger.
- Per-source log tagging
- Log levels: DEBUG, INFO, WARN, ERROR
- Writes to stdout (daemon mode) and per-source files in logs/

**9. Implement `bubblefish daemon` command:**
- Call NewDaemon(configDir)
- Set up SIGTERM/SIGINT handler for graceful shutdown
- Call daemon.Run()

**10. Signal handling for graceful shutdown:**
- On SIGTERM or SIGINT:
  1. Log "Shutting down..."
  2. Stop accepting new HTTP requests
  3. Drain all queues (with drain_timeout_secs from config)
  4. Close WAL (fsync + close)
  5. Close all destination connections
  6. Exit 0

### ACCEPTANCE CRITERIA
- [ ] `bubblefish daemon` starts and listens on configured port
- [ ] curl POST to /inbound/claude with valid API key returns 200 + payload ID
- [ ] Payload appears in SQLite database after write
- [ ] curl POST with invalid API key returns 401
- [ ] curl POST with oversized body returns 413
- [ ] curl POST when queue full returns 429 with Retry-After header
- [ ] Duplicate idempotency_key returns 200 without writing again
- [ ] GET /query/sqlite returns stored data with response filter applied
- [ ] GET /api/status returns valid JSON with source and destination stats
- [ ] Transform: trim() removes whitespace, coalesce() provides defaults
- [ ] SIGTERM triggers graceful shutdown — all queued items drain before exit
- [ ] Rate limiting returns 429 after exceeding per-source limit
- [ ] Per-source log files created in logs/ directory

### DO NOT
- Do not implement TUI or Web UI
- Do not implement hot-reload
- Do not implement remaining destination plugins (OpenBrain, ChromaDB, Postgres)
- Do not implement Prometheus metrics endpoint yet

## PHASE 4: REMAINING DESTINATIONS + OBSERVABILITY + DOCTOR
**Duration:** 3-4 days | **Depends on:** Phase 3

### CONTEXT
Phases 0-3 complete. Working daemon with HTTP server, write path (mapping → transform → idempotency → WAL → queue → SQLite), read path (query → filter → response), auth, rate limiting, graceful shutdown. SQLite is the only working destination.

### DELIVERABLES

**1. internal/destination/openbrain.go:** OpenBrain/Supabase destination.
- Uses net/http to call Supabase REST API
- Write(): POST to /rest/v1/{table} with payload as JSON
- Read(): GET from /rest/v1/{table} with query params
- HealthCheck(): HEAD request to Supabase URL
- Auth via apikey header from SupabaseKey config

**2. internal/destination/chromadb.go:** ChromaDB destination.
- Uses net/http to call ChromaDB REST API
- Write(): POST to /api/v1/collections/{collection}/upsert with documents + embeddings
- Read(): POST to /api/v1/collections/{collection}/query with query_embeddings
- Validates that payload.Vector is non-empty before writing (returns ClassifiedError{Class: ErrorPermanent} if empty)
- HealthCheck(): GET /api/v1/heartbeat

**3. internal/destination/postgres.go:** PostgreSQL destination.
- Uses jackc/pgx/v5 with connection pool
- Init(): create table if not exists (same schema as SQLite)
- Validates DSN contains sslmode parameter (redundant check — build should catch this, but defense in depth)
- Write(): INSERT with ON CONFLICT DO UPDATE
- Read(): SELECT with parameterized queries (no SQL injection)
- HealthCheck(): pool.Ping()
- Close(): pool.Close()

**4. internal/observability/metrics.go:** Prometheus metrics.
- Register metrics: bubblefish_requests_total (counter, labels: source), bubblefish_queue_depth (gauge, labels: destination), bubblefish_write_latency_seconds (histogram, labels: destination), bubblefish_errors_total (counter, labels: source, class), bubblefish_wal_entries_total (counter), bubblefish_wal_segment_size_bytes (gauge)
- Serve /metrics endpoint on metrics_addr from daemon config
- Instrument: handlers.go (request counter, latency), worker.go (write latency, error counter), wal.go (entry counter, segment size)

**5. internal/doctor/doctor.go:** Health check command.
```go
func CheckAll(configDir string) []CheckResult
```
- Check each destination is reachable (call Init + HealthCheck + Close)
- Check WAL directory is writable (create temp file, write, delete)
- Check disk space on WAL partition (warn if < 1 GB free)
- Check listen port is available (attempt bind, release)
- Check metrics port is available
- Check all API keys in source configs are non-empty and unique
- Print results with ✓/✗ indicators

**6. Implement `bubblefish doctor` command** using doctor.CheckAll().

**7. Update daemon.go** to instantiate all four destination types based on config.Type field.

## PHASE 5: TUI + WEB UI + HOT-RELOAD
**Duration:** 3-4 days | **Depends on:** Phase 4

### CONTEXT
Phases 0-4 complete. Full working daemon with all 4 destinations, Prometheus metrics, doctor command, auth, rate limiting, transforms, response filter, graceful shutdown. No visual interface yet. No hot-reload.

### DELIVERABLES

**1. Bubble Tea TUI (internal/tui/):**
- Polls /api/status every 2 seconds
- Polls /api/logs every 2 seconds
- Layout:
  - Header: "BubbleFish v1.0.0 | Uptime: Xh Xm | [destinations: ● sqlite ● postgres ○ chromadb]" (green = healthy, red = unhealthy)
  - Queue bar: "Queue: 3 waiting | 1 processing | 1,234 completed | 0 errors"
  - Traffic feed: scrolling list of recent events, color-coded by source name (claude=purple, chatgpt=green, perplexity=blue, ollama=yellow, lmstudio=cyan)
  - Footer: "[Q]uit [D]etails [L]ogs [R]efresh"
- Keyboard: q=quit, d=toggle detail pane (source/destination drill-down), l=toggle log view
- Connects to /api/status and /api/logs using the api_token from daemon.toml

**2. Web UI (web/):**
- Single HTML file with embedded CSS and JS (no build step, no npm)
- Served from Go's embed.FS
- Three sections: queue stats cards at top, source/destination table in middle, recent events table at bottom
- Auto-refresh every 5 seconds via fetch() to /api/status and /api/logs
- No admin controls (read-only)
- Clean, minimal design. Dark header, light body.

**3. internal/daemon/hotreload.go:** Atomic hot-reload.
- Polling-based config watcher (check file modification times every hot_reload_secs)
- On change detected:
  1. Re-run Build(configDir) with the new config files
  2. If build succeeds: stage the new CompiledConfig
  3. Swap the daemon's config pointer atomically (sync.Mutex)
  4. Log "Config reloaded successfully"
  5. DO NOT restart queues or destinations — only config/mapping/transform/filter rules change
  6. If build fails: log detailed error, keep previous config, continue running

**4. Simple agentic chaining (internal/daemon/chain.go):**
- If source config has chain_to set:
  1. After mapping + transform, forward the TranslatedPayload to the chained source's inbound endpoint (internal HTTP call)
  2. Wait for response (with configurable timeout, default 30s)
  3. If chain succeeds: use the chained response as the final payload for WAL + queue
  4. If chain fails: return error to original caller, do NOT write to WAL
- WAL captures BOTH the original payload AND the chained result (two entries, linked by a chain_id in metadata)

**5. Implement `bubblefish ui` command** — launches TUI.

**6. Update `bubblefish daemon`** to start hot-reload watcher and serve Web UI.

### ACCEPTANCE CRITERIA
- [ ] TUI displays live data from running daemon
- [ ] TUI color-codes source names correctly
- [ ] TUI keyboard shortcuts work (q, d, l)
- [ ] Web UI loads in browser at web_bind_addr
- [ ] Web UI auto-refreshes and shows current queue stats
- [ ] Hot-reload: edit a source TOML, daemon picks up changes within hot_reload_secs
- [ ] Hot-reload: invalid TOML edit does NOT crash daemon, logs error, keeps old config
- [ ] Agentic chain: source A → source B → destination works end-to-end
- [ ] Agentic chain: failure in chain returns error to caller, nothing written to WAL

## PHASE 7: TESTING + HARDENING + SHIP

### DELIVERABLES

- End-to-end: init → build → daemon → POST → verify in SQLite → GET → verify response filter

- Circuit breaker: make destination unreachable, verify circuit opens, make reachable again, verify recovery
- Rate limiting: exceed rate limit, verify 429 with Retry-After
- Schema version: create WAL entries with "0.9.0" schema, verify replay logs warning and skips

- [ ] No hardcoded secrets in source code

**3. Performance baseline:**
- Measure: sustained throughput (payloads/second) to SQLite on commodity hardware
- Measure: p99 write latency under load

**4. Final cleanup:**
- Remove all TODO comments or convert to GitHub issues

**5. Tag and release:**

- Write release notes

## NOTES FOR AI TOOL USAGE

**Token conservation strategy:**
- Hand each phase as a single prompt with the CONTEXT, DELIVERABLES, and ACCEPTANCE CRITERIA sections
- Include the Technical Spec (TECHNICAL_SPEC.md) as an attachment or reference file for Phases 0-3
- For Phases 4-7, include only the relevant sections of the Technical Spec
- At the start of each phase, tell the AI: "The following files already exist:" and list the key files from previous phases
- Do NOT paste all previous code into the prompt — just list file names and summarize what they contain

**If the AI runs out of context:**
- Split the phase into sub-phases (e.g., Phase 3a: Daemon + HTTP, Phase 3b: Transforms + Auth)
- Each sub-phase should still have its own acceptance criteria

**Quality gates between phases:**
- `go build ./...` must succeed
- `go vet ./...` must pass
- All acceptance criteria for the phase must be met before starting the next phase
- If a phase produces bugs, fix them BEFORE moving to the next phase
