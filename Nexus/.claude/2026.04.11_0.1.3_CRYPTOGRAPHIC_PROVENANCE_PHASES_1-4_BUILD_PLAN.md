## 2. Phase 1 — Foundation Layer

**Goal:** Batched group commit on WAL writes, incremental replay with consistent checkpoints, audit log as a first-class WAL entry type, bytes/sec rate limiting.

**Estimated subtasks:** 4. Estimated duration: 2-3 weeks at focused pace.

### 2.1 Group commit ring buffer (subtask 1.1)

**File to create:** `internal/wal/groupcommit.go`

**Design:**
- Single goroutine consumes from a buffered channel (`submitCh chan *pendingEntry`).
- Each `pendingEntry` carries the WAL entry bytes, a `done chan error`, and metadata.
- The consumer goroutine buffers up to `group_commit_max_batch` entries OR waits until `group_commit_max_us` elapsed since the first entry arrived, whichever first.
- On flush: write all buffered entries to the current WAL segment sequentially, call `file.Sync()` once, then send `nil` on every `done` channel. If sync fails, send the error on every channel.
- The HTTP handler (in `internal/daemon/handlers.go`) constructs a `pendingEntry`, sends it to `submitCh`, then blocks on `<-done`. Only after `done` receives `nil` does the handler return `200 OK`. Per-request durability is preserved because `200 OK` happens *after* the fsync completes.

**Config additions** (`daemon.toml`):
```toml
[daemon.wal]
# existing fields preserved
group_commit_enabled = true
group_commit_max_batch = 256
group_commit_max_us = 500
```

**Defaults:** `group_commit_enabled = true` on new installs, `false` on upgrades from v0.1.2 (preserve exact old behavior unless operator opts in). Add an `install --mode simple` change to write `true` for new installs.

**Test file to create:** `internal/wal/groupcommit_test.go`

**Required tests:**
1. Single writer, 1000 sequential writes, measure p50 and p99 latency. Must not regress by more than 10% vs v0.1.2 baseline.
2. 50 concurrent writers, 10,000 writes total. Must show at least 3× throughput improvement over v0.1.2 baseline.
3. Kill -9 mid-batch test: spawn a helper process that writes 500 entries, kill it at entry 250 via `taskkill /F`, restart daemon, verify entries that received `200 OK` are present and entries that did not are either present or absent (never partial/corrupted).
4. Deadline correctness: single write must never exceed `group_commit_max_us + 200μs` end-to-end latency under no-load conditions.
5. Buffer full correctness: fill the buffer to `max_batch`, verify flush fires immediately without waiting for deadline.
6. Empty buffer correctness: no entries submitted for 10 seconds, consumer goroutine blocks cleanly, no busy loop, no CPU use.

**Commit message:** `Phase1.1: group commit ring buffer with deadline-driven flush`

### 2.2 Incremental replay with consistent checkpoints (subtask 1.2)

**Files to modify:** `internal/wal/segment.go`, `internal/wal/replay.go` (create if missing), `internal/wal/checkpoint.go` (create)

**Design:**
- Add new WAL entry type `EntryTypeCheckpoint` alongside existing `EntryTypeData`.
- Checkpoint entry content: `{offset_bytes, timestamp, applied_count, state_hash}` where `state_hash` is SHA-256 of the primary destination table's contents at checkpoint time.
- Checkpoint markers are written inside the group commit batch every `checkpoint_interval_entries` (configurable, default 10,000).
- On startup, scan backward from the end of the WAL to find the most recent valid checkpoint. "Valid" means: CRC32 passes AND the computed state_hash matches the recorded state_hash.
- If a valid checkpoint is found, replay only entries after that offset.
- If no valid checkpoint is found, fall back to full replay from genesis (same behavior as v0.1.2).

**Config additions:**
```toml
[daemon.wal]
checkpoint_interval_entries = 10000
```

**Test file to create:** `internal/wal/checkpoint_test.go`

**Required tests:**
1. Write 50,000 entries, verify at least 4 checkpoint markers present in WAL.
2. Cold start with 100,000 entries + recent checkpoint: replay time must be sub-second on a consumer SSD.
3. Cold start with corrupted state_hash on most recent checkpoint: daemon falls back to previous checkpoint, logs warning, does not crash.
4. Cold start with zero valid checkpoints: daemon performs full replay, completes successfully, logs info message.
5. Kill -9 during checkpoint write: restart must either have the checkpoint or not have it, never a partial checkpoint.

**Commit message:** `Phase1.2: incremental WAL replay with consistent checkpoints`

### 2.3 Audit log as WAL entry type (subtask 1.3)

**Files to modify:** `internal/wal/segment.go`, `internal/audit/writer.go`, `cmd/bubblefish/audit.go` (create for the export command)

- `internal/audit/writer.go` is refactored: instead of writing JSONL directly, it constructs an `AuditEntry` and submits it to the WAL group commit ring buffer via a new `audit.Submit()` function.

- New CLI command `bubblefish audit export --format jsonl --since <ISO timestamp> --until <ISO timestamp>` reads audit entries from the WAL and emits JSONL to stdout.

**Test file to create:** `internal/audit/wal_audit_test.go`

**Required tests:**

2. Kill -9 between audit write and JSONL write: audit entry must be present in WAL on restart.
3. `bubblefish audit export` returns entries in time order.
4. Cross-check: for every audit entry in `security-events.jsonl`, there is a matching WAL audit entry.

**Commit message:** `Phase1.3: audit log as WAL entry type with group commit durability`

## 3. Phase 2 — Trust Boundary Layer

**Goal:** Tier partitions on sources, review token classes, per-tier rate limiting, embedding provider validation envelope.

**Estimated subtasks:** 4. Estimated duration: 2-3 weeks.

### 3.1 Tier partitions on sources (subtask 2.1)

**Design:**
- Add `tier` field (int 0-3) to source TOML. Default 1 (matches v0.1.2 behavior).
- Tier 0 = quarantined/untrusted. Tier 3 = fully trusted. Tiers 1 and 2 are intermediate.
- Entries are tagged with their source's tier at write time and stored in a new `tier` column on the entries table.
- On read, the retrieval cascade filters out entries whose tier is above the reading source's tier. Default rule: source at tier N can read entries at tier ≤ N.
- **Response-shape indistinguishability:** the response format must be identical whether filtering removed entries or not. Pad timing with `time.Sleep` if needed to prevent timing side-channels leaking "there exist entries above your tier."

**Files to modify:** `internal/config/source.go`, `internal/storage/sqlite.go` (add tier column + migration), `internal/retrieval/cascade.go`

**Migration:** On first v0.1.3 daemon start, add `tier INTEGER DEFAULT 1` column to the entries table. All existing entries get tier = 1. Non-destructive.

**Commit message:** `Phase2.1: tier partitions on sources with cross-tier filtering`

### 3.2 Review token classes (subtask 2.2)

**Design:**
- Add two new token prefixes to `internal/auth/tokens.go`: `bfn_review_list_` and `bfn_review_read_`.
- `bfn_review_list_` tokens can call `GET /api/review/queue` to list quarantined memory IDs and metadata, but cannot retrieve content.
- `bfn_review_read_` tokens can call `GET /api/review/{memory_id}` to read content of specific memory IDs.
- Both return 401 on any other endpoint.
- Both use constant-time comparison like existing token classes.
- These tokens will be used by the Phase 5 governance UI (not in this build). Phase 2 just establishes the namespaces.

**Commit message:** `Phase2.2: review token classes for governance workflow`

### 3.3 Per-tier rate limiting (subtask 2.3)

**Design:**
- Extend the rate limiter from Phase 1.4 to support per-tier limits via `[daemon.tiers.N]` blocks:
  ```toml
  [daemon.tiers.0]
  rate_limit_rpm = 60
  rate_limit_bytes_per_second = 65536

  [daemon.tiers.3]
  rate_limit_rpm = 10000
  rate_limit_bytes_per_second = 10485760
  ```
- Per-source limits override per-tier limits when both are set.

**Commit message:** `Phase2.3: per-tier rate limiting`

### 3.4 Embedding provider validation envelope (subtask 2.4)

**This is the novel piece nobody else has. Take extra care.**

**Design:**
- Every embedding returned by a provider is validated before being cached:
  1. **Shape check:** vector length matches expected dimensionality for the provider+model.
  2. **Content-hash integrity:** input text is hashed (SHA-256) and included in the cache key. Cache hits require both input_hash and provider_identity match.
  3. **Provider-identity stamping:** every cached embedding stores `provider_identity` (e.g., `ollama:nomic-embed-text@v1.5`).
  4. **Drift detection:** maintain running statistics (mean, variance per dimension) over the first 1000 embeddings from each provider_identity. Any embedding whose statistics deviate beyond 3 standard deviations on more than 10% of dimensions is flagged. Repeated flags (3 in a row) auto-quarantine the provider_identity (refuse to cache embeddings from it until an operator clears the quarantine).
- Add new endpoint `GET /api/embedding/providers` that lists known provider identities, their status (active/quarantined), and their drift statistics.
- Add CLI command `bubblefish embedding clear-quarantine <provider_identity>`.

**Files to create/modify:** `internal/embedding/validator.go`, `internal/embedding/drift.go`, `internal/embedding/cache.go`

**Tests:** verify drift detection catches a synthetic drifted embedding, verify quarantine blocks cache writes, verify operator clear-quarantine works.

**Commit message:** `Phase2.4: embedding provider validation envelope with drift detection`

## 4. Phase 3 — Cluster Mechanism

**Goal:** Write-time near-duplicate detection via LSH prefilter, cluster membership as indexed columns, async cluster assignment, cluster surfacing on read.

**Estimated subtasks:** 4. Estimated duration: 3-4 weeks.

### 4.1 LSH prefilter (subtask 3.1)

**Design:**
- Implement SimHash variant with random hyperplanes in `internal/cluster/lsh.go`.
- 16 random hyperplanes generated at daemon start from a seed stored in `daemon.toml` (`[daemon.cluster] lsh_seed = "..."`). Seed must be stable across restarts or bucket assignments become inconsistent.
- For each new embedding, compute its bucket ID: 16-bit integer where bit N is sign of dot product with hyperplane N.
- LSH is invoked on the write path but only to compute the bucket ID (a handful of dot products). Actual cluster assignment happens in the queue worker, off the write path.

**Commit message:** `Phase3.1: SimHash LSH prefilter for cluster bucketing`

### 4.2 Cluster membership columns (subtask 3.2)

**Design:**
- Add columns to entries table: `cluster_id TEXT`, `cluster_role TEXT` (one of `primary`, `member`, `superseded`), `lsh_bucket INTEGER`.
- Indexes: `(cluster_id, cluster_role)`, `(lsh_bucket)`.
- Migration: existing rows get `cluster_id = NULL`, `cluster_role = NULL`, `lsh_bucket = NULL`. Non-destructive.

**Commit message:** `Phase3.2: cluster membership columns with indexes`

### 4.3 Async cluster assignment (subtask 3.3)

**Design:**
- Queue worker, after destination write succeeds, does cluster assignment:
  1. Look up candidates in same `lsh_bucket`.
  2. Compute actual cosine similarity against each candidate using the full embedding.
  3. If similarity ≥ `cluster_threshold` (configurable, default 0.92) against any existing primary, assign to that cluster as `member`.
  4. Otherwise create a new cluster with this entry as `primary`, `cluster_id = <UUID>`.
  5. Cluster size cap: max `cluster_max_members` per primary (default 16). When cap is reached, overflow members go to a sibling cluster (same LSH bucket, different primary). Deterministic: sort by entry timestamp, first N stay in original, rest go to sibling.
  6. Clusters never span tiers. Each cluster is tier-isolated.

**Commit message:** `Phase3.3: async cluster assignment in queue worker`

### 4.4 Cluster surfacing on read (subtask 3.4)

**Design:**
- New retrieval profile `cluster-aware` added alongside `fast`, `balanced`, `deep`.
- `cluster-aware` profile: after Stage 5 hybrid merge, for each returned entry that has a `cluster_id`, fetch all members of that cluster and include them in the response.
- Conflict detection: if cluster members have different content hashes, the response metadata includes `_nexus.conflict = true` with an array of conflicting entries.
- Default profile remains `balanced` — no change in behavior unless client opts into `cluster-aware`.

**Commit message:** `Phase3.4: cluster surfacing on read with conflict detection`

- [ ] All existing tests still pass

- [ ] Cluster surfacing tests show both members returned for a conflicting cluster

## 5. Phase 4 — Provenance Layer + Cryptographic Demo

**Goal:** Per-source Ed25519 keypairs, signed write envelopes, hash-chained audit log, verify endpoint and CLI, 60-second cross-vendor demo.

**Estimated subtasks:** 5. Estimated duration: 3-4 weeks.

### 5.1 Per-source Ed25519 keypairs (subtask 4.1)

**Config addition** (per source TOML):
```toml
[source.signing]
mode = "local"
key_file = "~/.bubblefish/Nexus/sources/claude.key"
```

Modes: `none` (default, no signing), `local` (key on disk, signed locally), `external` (stub — returns "not yet implemented" error if configured, deferred to future version).

On first daemon start with `mode = local` and missing `key_file`: generate Ed25519 keypair, write private key with 0600 permissions. Public key stored alongside with `.pub` suffix.

New CLI commands:
- `bubblefish source rotate-key <name>` — archive old key, generate new one
- `bubblefish source pubkey <name>` — print public key in base64

**Commit message:** `Phase4.1: per-source Ed25519 keypairs with three attestation modes`

### 5.2 Signed write envelopes (subtask 4.2)

**Design:**
- When a source with `mode = local` writes a memory:
  1. Compute content hash: `SHA-256(canonical(payload))` where canonical means sorted keys, no whitespace.
  2. Signing payload: `"${source_name}|${timestamp}|${idempotency_key}|${content_hash}"` as ASCII bytes.
  3. Signature: Ed25519 sign the payload with the source's private key.
  4. Store signature in new `signature` column (base64) and `signing_key_id` column (fingerprint of public key).
- Entries without signatures continue to work (backward compat).
- New endpoint `GET /api/sources/{name}/pubkey` returns the public key.

**Commit message:** `Phase4.2: signed write envelopes`

### 5.3 Hash-chained audit log (subtask 4.3)

**Design:**
- Genesis audit entry written at first daemon startup: contains daemon identity (hostname, install timestamp, daemon Ed25519 public key).
- Every audit entry's serialized content includes `prev_audit_hash` (SHA-256 of the previous entry's serialized content).
- Chain extension happens inside the group commit batch from Phase 1.3 — same fsync durability as data writes.
- Single-writer constraint: chain extension is serialized through the group commit goroutine. No concurrent chain extension.
- On replay: verify each audit entry's `prev_audit_hash` matches the previous entry. If mismatch detected, fail closed: refuse to start the daemon and require operator intervention (log clear error, exit code 2).

**Commit message:** `Phase4.3: hash-chained audit log with fail-closed integrity verification`

### 5.4 Verification endpoint and CLI (subtask 4.4)

**New endpoint:** `GET /verify/{memory_id}` returns:
```json
{
  "memory": { /* full entry */ },
  "signature": "base64...",
  "source_pubkey": "base64...",
  "signing_key_id": "fingerprint",
  "audit_entries": [ /* chain from genesis to this memory's audit entry */ ],
  "daemon_pubkey": "base64...",
  "genesis_entry": { /* for root-of-trust */ }
}
```

**New CLI command:** `bubblefish verify <proof-file.json>`

The verify CLI:
1. Loads the proof bundle
2. Verifies the source signature against the source's public key (`crypto/ed25519.Verify`)
3. Recomputes the audit chain from genesis forward, checking every `prev_audit_hash`
4. Verifies the memory's audit entry is correctly chained in
5. Prints a human-readable report: valid/invalid, which check passed/failed, why
6. Exit code 0 on valid, 1 on invalid

**The verify CLI must not require a running Nexus** — it operates purely on the proof file. This is the property that makes the proof useful for compliance audits.

**Commit message:** `Phase4.4: verification endpoint and standalone verify CLI`

### 5.5 The 60-second cross-vendor demo (subtask 4.5)

**Create directory:** `examples/cryptographic-provenance/`

**Files to create:**
- `README.md` — the demo script as a narrative
- `demo.sh` (Linux/macOS) and `demo.ps1` (Windows) — automated demo
- `agent-a.toml` — source config with signing enabled
- `agent-b.toml` — source config with signing enabled

**Demo flow (automated by the script):**
1. Install both source configs
2. Start daemon
3. Write a memory from `agent-a` containing: `"The project codename is Blue Ocean."`
4. Read it from `agent-b` via `nexus_search` — verify it returns
5. Export proof bundle: `curl http://localhost:8080/verify/<memory_id> > proof.json`
6. Run `bubblefish verify proof.json` — succeeds with exit code 0, prints a "VALID" report
7. Directly modify the SQLite file: `sqlite3 nexus.db "UPDATE entries SET content='The project codename is Red Mountain.' WHERE id=...;"`
8. Run `bubblefish verify proof.json` again — fails with exit code 1, prints an "INVALID: signature mismatch" report
9. Clean up and print summary: "Write from agent-a → read from agent-b → proof verified → database tampered → verification failed. 60 seconds."

**This is the demo that ends up on HN.** Polish the output. Make it beautiful.

**Commit message:** `Phase4.5: 60-second cross-vendor cryptographic provenance demo`

Stop immediately. Do not proceed. This is a hard constraint violation. Revert the offending commit and find a backward-compatible approach.

**If a test starts flaking:**
Do not skip it. Do not mark it as expected-fail. Find the root cause before moving on. Flaky tests compound into unshippable releases.

**If the group commit work reveals a fundamental issue with the existing WAL design:**
Stop and notify Shawn. Do not attempt a unilateral architectural redesign. The WAL is the product's central durability claim.
