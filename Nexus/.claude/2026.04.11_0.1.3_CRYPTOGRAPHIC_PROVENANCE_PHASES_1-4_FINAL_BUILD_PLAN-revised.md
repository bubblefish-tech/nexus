## 0. Working principles (non-negotiable)

1. **Reliability first. Speed second. Features third.** This is the project design philosophy. When any two conflict, reliability wins.
2. **WAL first, queue second, destination third. ALWAYS.** No shortcuts.
3. **Backward compatibility is mandatory.** v0.1.2 WAL files, source configs, destination configs, and tokens must continue to work unchanged. Sources without new config blocks behave exactly like v0.1.2.
4. **Every WAL change requires a kill -9 test.** 100 consecutive iterations minimum.
5. **The system fails closed, not open.** Every error path defaults to refusing the operation rather than allowing it.
6. **Any optimization that weakens durability gets rejected.** No exceptions.
7. **All token comparisons: `crypto/subtle.ConstantTimeCompare`.**
8. **Secret hygiene.** Never real tokens, real keys, or real Cloudflare hostnames in any file. Use `bfn_data_TEST_KEY`, `bfn_mcp_TEST_KEY`, `ed25519_test_pubkey_PLACEHOLDER` in fixtures.
9. **Canonical domain: `bubblefish.sh`.** Never `bubblefish.ai`.
10. **Copyright header on every new `.go` file:** `// Copyright © 2026 BubbleFish Technologies, Inc.`
11. **No patent language. No named competitors. Altman doctrine.**
12. **Quality gate after every subtask:**
    ```powershell

    $env:CGO_ENABLED='1'; go test ./... -race -count=1
    ```

## 2. Phase 1 — Foundation Layer (hardened)

### 2.1 Group commit ring buffer (subtask 1.1)

**File:** `internal/wal/groupcommit.go`

**Design:**
- Single consumer goroutine reads from `submitCh chan *pendingEntry`
- Batches up to `group_commit_max_batch` entries OR until `group_commit_max_us` elapsed since first entry
- On flush: write all entries, call `file.Sync()` once, signal every `done` channel
- `done` signaling MUST happen only after `file.Sync()` returns nil

**Config:**
```toml
[daemon.wal]
group_commit_enabled = true
group_commit_max_batch = 256
group_commit_max_us = 500
```

**Runtime mode flip:** `bubblefish wal set-mode legacy|group-commit` command that changes behavior without restart. Escape hatch if bug ships.

**Tests:** 1000 sequential writes latency, 10000 concurrent writes throughput, kill -9 mid-batch (100 iterations), deadline correctness, buffer full correctness, empty buffer (no busy loop).

### 2.2 Dual integrity: CRC32 + 8-byte sentinels (subtask 1.2)

Every WAL entry gets `0xBFBFBFBF` sentinel at start and `0xFBFBFBFB` at end in addition to CRC32. Torn sector writes that happen to preserve CRC bytes are caught by sentinel mismatch.

### 2.3 Incremental replay with consistent checkpoints (subtask 1.3)

Simple rule: a checkpoint is valid IFF all three checks pass (CRC32, state_hash, applied_count). Any failure → full genesis replay. No fallback-to-previous-checkpoint logic.

Checkpoint content: `{offset_bytes, timestamp, applied_count, state_hash, audit_chain_hash}`.

### 2.4 Audit log as WAL entry type (subtask 1.4)

New `EntryTypeAudit`. `internal/audit/writer.go` submits to group commit ring buffer. `security-events.jsonl` becomes a derived tail-follower. New CLI `bubblefish audit export --format jsonl --since <ts> --until <ts>`.

### 2.5 Bytes/sec rate limiting (subtask 1.5)

`golang.org/x/time/rate` token bucket, per source. Per-source config `rate_limit_bytes_per_second`. Counts full request body. Distinct 429 code: `rate_limit_bytes_exceeded`.

### 2.6 fsync verification on startup (subtask 1.6)

`bubblefish doctor --fsync-test` writes, fsyncs, reads back via fresh fd, compares. On daemon startup, same test runs once and logs a warning if fsync appears to be a no-op (broken on network storage, some consumer SSDs).

### 2.7 Disk-full reservation check (subtask 1.7)

Before starting a group commit batch, verify `(max_batch × max_entry_size)` bytes available on WAL filesystem. If not, reject writes with 503 until space freed. Pre-allocate next segment at 80% fill.

### 2.8 Goroutine heartbeat supervisor (subtask 1.8)

Every critical goroutine (group commit consumer, queue workers, audit writer) sends heartbeat on shared channel every 1 second. Supervisor goroutine monitors. If any heartbeat silent for 30 seconds, dump all goroutine stacks to `logs/deadlock-<timestamp>.log`, log fatal, exit with code 3. Systemd/launchd restarts. WAL replay handles in-flight state.

### 2.9 Monotonic clock for ordering (subtask 1.9)

All ordering-sensitive operations use `time.Now()` which includes monotonic component. Wall clock recorded separately for display. Clock skew, NTP corrections, VM migrations cannot break ordering invariants.

### 2.10 WAL compression via zstd (subtask 1.10)

WAL entries compressed with `github.com/klauspost/compress/zstd` before write. 3-5x size reduction, fewer bytes to fsync, smaller backups. Compression happens before CRC32 computation.

### 2.11 Phase 1 exit criteria

- All subtasks committed, all tests pass, 100× kill -9 passes
- Throughput ≥ 3× v0.1.2 baseline
- v0.1.2 WAL files load cleanly
- `CHANGELOG.md` and `CONFIGURATION.md` updated

## 3. Phase 2 — Trust Boundary Layer

### 3.1 Tier partitions with SQL-layer enforcement (subtask 2.1)

`tier` column (int 0-3) on entries, default 1. Non-destructive migration. **Enforcement in the SQL WHERE clause**, not post-filter: `AND tier <= ?` with requesting source's tier as parameter. SQLite never touches rows above the tier — closes timing side channel.

### 3.2 LSH tier-scoped buckets (subtask 2.2)

LSH seed is per-tier, not global. Same content in different tiers produces different bucket IDs. Cross-tier bucket collision impossible by construction.

### 3.3 Review token classes (subtask 2.3)

`bfn_review_list_` (list quarantined memory IDs), `bfn_review_read_` (read content of specific IDs). Both constant-time compared. Both 401 on any other endpoint. For Phase 5 governance UI.

### 3.4 Per-tier rate limiting (subtask 2.4)

`[daemon.tiers.N]` blocks with `rate_limit_rpm` and `rate_limit_bytes_per_second`. Per-source overrides per-tier.

### 3.5 Embedding validation envelope with fresh baselines (subtask 2.5)

Shape check, content-hash integrity, provider-identity stamping, drift detection with 3-sigma threshold. **New provider identity = fresh baseline rule** — first 1000 embeddings establish baseline without triggering drift alarm. Quarantine state stored in WAL, not SQLite.

### 3.6 Secrets directory (subtask 2.6)

`~/.bubblefish/Nexus/secrets/` with 0700 dir perms. LSH seed, Ed25519 keys (Phase 4), future HMAC keys all live here. Separate from config files so TOML files can be checked into version control without leaking secrets.

### 3.7 Phase 2 exit criteria

All subtasks committed, tests pass, **pause for overseer checkpoint**.

## 4. Phase 3 — Cluster Mechanism

### 4.1 SimHash LSH prefilter (subtask 3.1)

16 hyperplanes per tier, seeds in `secrets/lsh-tier-N.seed`. Bucket ID = 16-bit integer from sign of dot products.

### 4.2 Cluster columns (subtask 3.2)

`cluster_id TEXT`, `cluster_role TEXT` (primary|member|superseded), `lsh_bucket INTEGER`. Indexes on `(cluster_id, cluster_role)` and `(lsh_bucket, tier)`. Non-destructive migration.

### 4.3 Async cluster assignment (subtask 3.3)

Queue worker computes bucket candidates, cosine similarity, assigns to existing cluster if similarity ≥ 0.92 else creates new. Cluster size cap 16, deterministic sibling overflow by timestamp. Clusters never span tiers.

Cluster role is display convention only. Each entry keeps its original signature regardless of cluster membership changes.

### 4.4 Cluster surfacing with conflict detection (subtask 3.4)

New retrieval profile `cluster-aware`. Default `balanced` unchanged. When enabled, returns all cluster members. Response metadata includes `_nexus.conflict = true` when members have different content hashes.

### 4.5 Phase 3 exit criteria

All subtasks committed, tests pass including contradictory-write conflict detection, **pause for overseer checkpoint**.

## 5. Phase 4 — Provenance + Cryptographic Demo + Phase 6 Minimal

### 5.1 Per-source Ed25519 keys (subtask 4.1)

`[source.signing] mode = "local"`. Keys in `secrets/sources/<name>.ed25519` at 0600. Config references by path. New CLIs: `bubblefish source rotate-key <name>`, `bubblefish source pubkey <name>`.

**Key rotation chain:** New keypair's first operation signs old public key fingerprint. Verifiable chain of key identities per source.

### 5.2 Signed write envelopes (subtask 4.2)

Payload: `{source_name | timestamp | idempotency_key | content_hash}`. Ed25519 signature stored in new `signature` and `signing_key_id` columns. Unsigned entries continue to work.

Signature field is versioned: `signature_alg = "ed25519"` — future post-quantum algorithms pluggable.

### 5.3 Hash-chained audit log (subtask 4.3)

Genesis entry at first daemon startup contains daemon identity. Each audit entry includes `prev_audit_hash`. Chain extension inside group commit batch — same fsync durability as data. Single-writer via group commit goroutine.

On replay: verify chain from genesis. Mismatch → fail closed with exit code 2 AND clear error pointing to `bubblefish audit recover`.

### 5.4 Audit recover command (subtask 4.4, MANDATORY)

`bubblefish audit recover` — inspect corrupted chain, identify corruption offset, produce forensic report, offer operator choice: (a) truncate chain past corruption with clear audit trail of what was truncated and why, (b) abort. Without this command, a single bit flip bricks the daemon permanently.

### 5.5 Automatic idempotency for MCP (subtask 4.5)

If MCP `nexus_write` call does not include `X-Idempotency-Key`, daemon generates one: `SHA-256(client_session_id || content || timestamp_second)`. Same call within same second from same client is idempotent. Network hiccup retries no longer create duplicates.

### 5.6 Verify endpoint and CLI (subtask 4.6)

`GET /verify/{memory_id}` returns proof bundle with memory, signature, source pubkey, signing key ID, audit entries from genesis to memory's audit entry, daemon pubkey, genesis entry.

`bubblefish verify <proof-file.json>` standalone CLI:
- Verifies source signature
- Recomputes audit chain from genesis
- Distinguishes "invalid signature" from "unknown daemon identity" from "audit chain mismatch"
- Exit 0 valid, 1 invalid
- No running daemon required

**Parallel chain verification:** fork-join over chunks. Large audit logs verify in seconds, not minutes.

**Reproducible build:** `-trimpath -buildid=` flags, published build script. Operators can rebuild verify CLI from source and get bit-identical binary.

### 5.7 Python verify implementation (subtask 4.7)

`tools/verify-python/verify.py` — ~200 lines of Python that does exactly what `bubblefish verify` does. Independent implementation proves the proof format is spec, not trick. Ships in the repo.

### 5.8 Minimal Phase 6: Daily Merkle root + external anchor (subtask 4.8)

At midnight UTC, compute Merkle root over the day's audit chain entries. Store in `data/merkle-roots/<YYYY-MM-DD>.json` signed with daemon key.

`bubblefish anchor setup --gist` — one-time setup. After setup, daily Merkle roots auto-publish to a public GitHub Gist via the GitHub API. External anchor that makes the audit chain verifiable against a public timestamp. Operators running this in production have tamper evidence that lives outside their own system.

### 5.9 Bubblefish prove query attestation (subtask 4.9)

`POST /api/prove` — takes a query, returns cryptographic proof of the result. Proof is (query hash, result set hash, daemon signature, audit chain extension). Every query becomes attestable.

### 5.10 Bubblefish timeline command (subtask 4.10)

`bubblefish timeline <memory_id>` — walks audit chain, returns full history of every version, read, cluster assignment, policy decision that touched the memory. Forensic history on demand.

### 5.11 Dashboard Proofs tab (subtask 4.11)

New tab in web dashboard. Live view of audit chain growing, current Merkle root, verification status of recent writes, proof export buttons. Visual demo that makes the feature obvious in 5 seconds.

### 5.12 The 60-second cross-vendor demo (subtask 4.12)

`examples/cryptographic-provenance/` with `README.md`, `demo.sh`, `demo.ps1`, `agent-a.toml`, `agent-b.toml`.

Demo flow: write from agent-a → read from agent-b → export proof → verify succeeds → tamper SQLite → verify fails → print "60 seconds, full cryptographic provenance across vendors."

This is what gets posted to HN.

### 5.13 Phase 4 exit criteria

All subtasks committed, cryptographic demo runs cleanly, Python verifier matches Go verifier on 100 proof bundles, **pause for overseer checkpoint**.

## 6. Viral features + testing infrastructure

### 6.1 `bubblefish chaos` command (subtask V.1)

Operator-runnable fault injection tool. Accepts an HTTP endpoint, sends workload, injects random faults (kill -9, fsync failures, disk full, partial writes, network drops). Measures data loss. Produces public-readable report.

Works against any Nexus instance including the operator's own. Anyone can verify the durability claim.

### 6.2 `bubblefish simulate` deterministic testing (subtask V.2)

FoundationDB-style simulation harness. Entire daemon runs against mock clock, mock filesystem, mock network. Every nondeterministic decision seeded. `bubblefish simulate --seed <N>` runs one scenario. CI runs millions per hour.

Bugs found by simulation are reported with seed numbers so anyone can reproduce. The tool's existence is the marketing.

### 6.3 `bubblefish sentinel` continuous drift detection (subtask V.3)

Background goroutine samples 1% of writes and reads for integrity verification. Alarms on any anomaly. Zero effect on hot path. Catches drift between WAL and destinations in real time.

### 6.4 Pluggable audit sinks (subtask V.4)

Native syslog, fluent-bit, OpenTelemetry sinks alongside existing webhook. Each ~half day of work. Enterprise deployable on day one.

### 6.5 Backup verify command (subtask V.5)

`bubblefish backup verify <path>` runs full audit chain verification and all checksum checks against a backup without restoring. Backups that can't be verified are theatre.

### 6.6 Destination rebuild command (subtask V.6)

`bubblefish destination rebuild <name>` replays WAL into fresh destination file. Documented recovery procedure for destination corruption.

### 6.7 OAuth routes unregistered when disabled (subtask V.7)

When `[daemon.oauth] enabled = false`, HTTP routes do not register. No disabled-code attack surface. Compile-time conditional via build tags considered overkill; runtime route registration is sufficient.

### 6.8 24-hour chaos run before tagging (subtask V.8)

Before section 7 release, run `bubblefish chaos` against a dedicated test daemon for 24 continuous hours. Any data loss or hang fails the release. Publish the test logs alongside the v0.1.3 release.
