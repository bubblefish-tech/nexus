## Step 8 — Chaos Scenarios

### Why deferred

The four chaos scenarios require subprocess spawning infrastructure (start a `bubblefish` process, kill it with SIGKILL at a precise moment, restart, verify invariants). This infrastructure does not exist in the codebase today. The existing `internal/chaos/` harness tests WAL durability via concurrent writers + random faults but does not spawn/kill subprocesses.

Building the subprocess harness is estimated at 4-6 hours and is better suited to a dedicated followup rather than blocking the v0.1.3 tag.

### What to build

**File:** `internal/substrate/chaos_test.go`
**Build tag:** `//go:build chaos` (excluded from default `go test`)

#### Scenario 8a — kill-9 during sketch write

**Setup:** Start daemon with substrate enabled. Begin writing a memory with a non-empty embedding.

**Kill point:** Between `destination.Write(tp)` succeeding and `substrate.ComputeAndStoreSketch()` completing. Implement via a test-only env var `BF_CHAOS_KILL_AT=sketch_write` that triggers `os.Exit(1)` (or `syscall.Kill(os.Getpid(), syscall.SIGKILL)`) inside the `onSubstrateWrite` callback.

**Post-kill verification:**
1. Restart daemon
2. Memory row exists in SQLite (destination write succeeded)
3. `sketch`, `embedding_ciphertext`, `embedding_nonce` columns are NULL (substrate write didn't complete)
4. `CuckooLookup(memoryID)` returns false (cuckoo insert didn't happen)
5. Audit log has the destination write but no `sketch_written` event
6. Memory is queryable via normal cascade (Stages 1-4) but not via sketch path

**Recovery:** The memory is in a consistent state — it simply has no substrate data. A future `bubblefish substrate backfill` command (v0.2) can recompute sketches.

#### Scenario 8b — kill-9 during ratchet advance

**Setup:** Start daemon with substrate enabled and at least one sketched memory.

**Kill point:** Between `INSERT INTO substrate_ratchet_states` (new state persisted) and `UPDATE substrate_ratchet_states SET state_bytes = zeroes` (old state shredded). Implement via `BF_CHAOS_KILL_AT=ratchet_advance_after_insert`.

**Post-kill verification:**
1. Restart daemon
2. DB has the new ratchet state row (INSERT committed)
3. Old state row may or may not be shredded (UPDATE may not have committed)
4. `RatchetManager.loadOrInitialize()` loads the latest non-shredded state
5. If both old and new states are non-shredded, the manager picks the newest one
6. Secrets file may contain old or new state (DB is ground truth)
7. Audit event for the advance is present (if WAL committed it)

**Recovery:** DB is ground truth. The manager loads the latest state. If the old state wasn't shredded, it remains in the table with `shredded_at = NULL` — this is safe because the new state is already active. A cleanup pass can shred orphaned old states on startup.

#### Scenario 8c — kill-9 during cuckoo persist

**Setup:** Start daemon with substrate enabled. Write several memories. Trigger a cuckoo `Persist()` call.

**Kill point:** During `db.Exec(INSERT INTO substrate_cuckoo_filter ...)`. Implement via `BF_CHAOS_KILL_AT=cuckoo_persist`.

**Post-kill verification:**
1. Restart daemon
2. `LoadCuckooOracle()` either loads the last successfully persisted filter, or fails
3. On failure, `RebuildFromDB()` scans the `memories` table and rebuilds
4. Rebuilt filter matches DB state — no false negatives on memories that existed before the crash
5. Audit log has a `cuckoo_rebuild` event if rebuild occurred

**Recovery:** Cuckoo filter is rebuilt from ground truth (memories table). This is O(n) but rare and happens only on startup after a crash during persist.

#### Scenario 8d — kill-9 during shred-delete

**Setup:** Start daemon with substrate enabled. Write a memory with a sketch. Begin a shred operation.

**Kill point:** Between `UPDATE memories SET embedding_ciphertext = NULL` and `cuckoo.Delete(memoryID)`. Implement via `BF_CHAOS_KILL_AT=shred_after_clear`.

**Post-kill verification:**
1. Restart daemon
2. Memory row has `embedding_ciphertext = NULL` and `embedding_nonce = NULL` (UPDATE committed)
3. But cuckoo filter still contains the memory ID (delete didn't happen)
4. This is a false positive in the cuckoo filter — harmless because:
   - Stage 3.5 will try to load the sketch, which may still exist (sketch column wasn't cleared)
   - Stage 4 full-precision rerank will not find the decryptable embedding (ciphertext is NULL)
   - The cascade degrades gracefully to non-substrate results
5. A startup reconciliation pass could detect this inconsistency:
   - Scan `substrate_memory_state` for entries whose memory has NULL ciphertext
   - Delete those entries from the cuckoo filter
   - This is a v0.1.4 enhancement

**Recovery:** The system reaches a safe state via the cascade's fail-open design. The false positive in the cuckoo filter is harmless. Full cleanup is a startup reconciliation pass (v0.1.4).

### Implementation approach

```go
// internal/substrate/chaos_test.go
//go:build chaos

package substrate

import (
    "os"
    "os/exec"
    "testing"
)

// chaosKillPoint checks if the current execution should be killed at this point.
// Used by production code only when BF_CHAOS_KILL_AT is set.
func chaosKillPoint(name string) {
    if os.Getenv("BF_CHAOS_KILL_AT") == name {
        os.Exit(137) // simulate SIGKILL
    }
}
```

Production code inserts `chaosKillPoint("sketch_write")` etc. at the appropriate locations, gated behind the env var check (zero cost when env var is unset).

Each test:
1. Builds the test binary
2. Spawns it as a subprocess with the appropriate `BF_CHAOS_KILL_AT` value
3. Waits for the process to exit (non-zero)
4. Opens the SQLite database directly
5. Verifies the invariants
6. Optionally restarts the daemon and verifies recovery

4-6 hours for all four scenarios including the subprocess harness.

- **Shred deletions**: Pick 3 memories at random intervals and run:
