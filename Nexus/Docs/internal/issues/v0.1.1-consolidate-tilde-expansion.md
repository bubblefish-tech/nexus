# Issue: Consolidate tilde expansion into a single shared helper

**Status:** Not started.
**Target:** v0.1.1
**Priority:** Low (code quality, not correctness)

## Background

The v0.1.0 path-handling audit identified four divergent tilde expansion
implementations across the codebase:

- `internal/daemon/daemon.go:1075-1083` — `expandPath` function
- `internal/daemon/daemon.go:424,466` — inline tilde handling with different slicing (`[2:]` instead of `[1:]`)
- `internal/backup/backup.go:500-508` — `expandPath` function
- `cmd/nexus/status.go:275-283` — `expandTilde` function (had a silent-error bug fixed in v0.1.0)

The functional differences between `[1:]` and `[2:]` are currently papered
over by `filepath.Join` normalization, but the duplication is fragile and
will drift over time.

## Proposed fix

1. Extract a single `ExpandTilde(p string) (string, error)` function in a
   shared package (probably `internal/config` or a new `internal/pathutil`).
2. Replace all four implementations with calls to the shared helper.
3. Add unit tests covering: tilde path, non-tilde path, empty string,
   bare `~`, `~/subdir`, `~\subdir` (Windows), error propagation when
   `os.UserHomeDir()` fails.

## Why this was deferred from v0.1.0

The duplication is cosmetic, not a correctness bug (all four
implementations produce correct results under normal conditions). The
one actual bug within this finding — the silent error swallow in
status.go — was fixed in v0.1.0. The broader refactor was deferred to
keep the launch-week scope small.
