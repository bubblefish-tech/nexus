# Issue: Apply filepath.ToSlash to loader path defaults

**Status:** Not started.
**Target:** v0.1.1
**Priority:** Low (latent trap, not a current bug)

## Background

The v0.1.0 path-handling audit identified an inconsistency between how
install generates TOML paths vs how the loader fills in defaults:

- `cmd/bubblefish/install.go` uses `filepath.ToSlash()` to write
  forward-slash paths into TOML files, avoiding backslash-escape issues
  on Windows.
- `internal/config/loader.go` uses plain `filepath.Join` for the WAL
  and audit log defaults, producing backslash paths on Windows.

These defaults are only filled in-memory during config load and never
written back to TOML, so the inconsistency does not cause a runtime
bug today. However, if a future code path serializes loader defaults
back to TOML (e.g., `bubblefish config export`, a config migration
tool, or a diagnostic dump), Windows backslashes would be
misinterpreted as TOML escape characters.

## Proposed fix

Apply `filepath.ToSlash()` to the path defaults in `loader.go` at
lines 161 (audit log) and 408 (WAL). Alternatively, document the
convention that in-memory paths are OS-native and must be converted
before serialization.

## Why this was deferred from v0.1.0

The auditor explicitly noted this is "not a runtime bug today." Fixing
it preemptively is the right long-term move but not a launch blocker.
