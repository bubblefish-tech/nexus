# Issue: Standardize slog error-key naming across the codebase

**Status:** Not started.
**Target:** v0.1.1
**Priority:** Low (affects log aggregation, not correctness)

## Background

The v0.1.0 path-handling audit noted inconsistent slog error-key naming.
Some calls use `"error"` as the key for error values; others use `"err"`.
Example: `internal/wal/updater.go:254` uses `"error"`, line 281 uses
`"err"`.

This affects log aggregation workflows: queries that filter on the error
key must search for both variants.

## Proposed fix

Standardize on one key (recommendation: `"error"`) across all slog calls
in the codebase. Grep for `slog.Error`, `slog.Warn`, `slog.Info` calls
that include an error value and normalize the key name. Add a linter
rule or CONTRIBUTING.md note to enforce the convention going forward.

## Why this was deferred from v0.1.0

Log key inconsistency does not affect correctness — only the ergonomics
of log aggregation queries. Users launching Nexus for the first time
will not set up log aggregation on day one. This is a v0.1.1 cleanup.
