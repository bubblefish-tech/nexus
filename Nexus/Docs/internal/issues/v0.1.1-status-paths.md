# Issue: Add `bubblefish status` subcommand with `--paths` flag

**Status:** Not started.
**Target:** v0.1.1
**Priority:** Medium

## Background

v0.1.0 added BUBBLEFISH_HOME and --home flag support to the install command,
config.ConfigDir(), and the generated daemon.toml. This enables sandbox
testing and CI isolation. However, there is no diagnostic tool that prints
the resolved data locations, so users debugging "where did my data go"
questions must grep TOML files manually.

## Proposed implementation

Add a `bubblefish status` subcommand with these features:

### `bubblefish status` (default behavior)
- Daemon health check (calls `/health` and `/ready` if daemon is running)
- Queue depth, WAL pending count, consistency score
- Last consistency check timestamp
- Cache hit rates
- Recent error count from security log

### `bubblefish status --paths`
- Print resolved config directory and all data file paths
- Format:
```
  bubblefish paths:
    config directory:    <path> (exists)
    daemon config:       <path>/daemon.toml (exists)
    sources directory:   <path>/sources/ (exists)
    destinations dir:    <path>/destinations/ (exists)
    wal:                 <path> (exists)
    sqlite database:     <path> (exists)
    security log:        <path> (does not exist)
    audit log:           <path> (exists)
```
- Each path is followed by `(exists)` or `(does not exist)` from `os.Stat`
- Works whether or not the daemon is running (reads config from disk only)
- Honors BUBBLEFISH_HOME like all other subcommands

### `bubblefish status --json`
- Machine-readable output of all of the above
- For CI integration and monitoring

## Why this matters

Users running Nexus in sandboxes (test environments, CI, multi-instance setups)
need a fast way to verify which config directory and data files the daemon is
actually using. Without this, debugging requires grepping TOML files and
correlating against `os.UserHomeDir()` behavior.

## Out of scope for v0.1.1

- Modifying daemon configuration via the status command
- Anything that requires the daemon to be running (those belong in admin endpoints)
- Health checks beyond what `/health` and `/ready` already report
